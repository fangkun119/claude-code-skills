# 四级标题体系（Title System）

## 概述

改写文档中所有章节标题必须严格遵循以下四级标题体系，不允许超出这四个层级。

## 标题层级规格

### Level 1：章（Chapter）

| 属性 | 值 |
|------|-----|
| 名称 | 章 |
| Markdown 前缀 | `##` |
| 编号模式 | 阿拉伯数字递增 |
| 格式 | `## {n}. ` |
| 示例 | `## 1. ` |

**完整示例**：
```markdown
## 1. 系统架构
```

---

### Level 2：节（Section）

| 属性 | 值 |
|------|-----|
| 名称 | 节 |
| Markdown 前缀 | `###` |
| 编号模式 | 父级.子级递增 |
| 格式 | `### {n}.{m} ` |
| 示例 | `### 1.1 ` |

**完整示例**：
```markdown
### 1.1 整体设计
```

---

### Level 3：小节（Subsection）

| 属性 | 值 |
|------|-----|
| 名称 | 小节 |
| Markdown 前缀 | `####` |
| 编号模式 | 圆括号阿拉伯数字递增 |
| 格式 | `#### ({k}) ` |
| 示例 | `#### (1) ` |

**完整示例**：
```markdown
#### (1) 模块划分
```

---

### Level 4：段落条目（Item）

| 属性 | 值 |
|------|-----|
| 名称 | 段落条目 |
| Markdown 前缀 | `#####` |
| 编号模式 | 带圈数字递增 |
| 格式 | `##### {circled_n} ` |
| 示例 | `##### ① ` |

**完整示例**：
```markdown
##### ① 用户界面层
```

---

## 完整示例

以下是一个符合四级标题体系的完整文档示例：

```markdown
## 1. 系统概述

本文档介绍了系统的整体架构和核心功能。

### 1.1 设计目标

系统设计的主要目标如下：

#### (1) 高可用性

##### ① 服务稳定性
确保服务 99.9% 的可用性。

##### ② 容错能力
支持故障自动转移。

#### (2) 可扩展性

##### ① 水平扩展
支持动态添加节点。

##### ② 垂直扩展
支持升级硬件配置。

## 2. 技术架构

### 2.1 分层设计

系统采用经典的分层架构。

#### (1) 表现层

##### ① Web 界面
提供基于浏览器的访问方式。

##### ② 移动应用
提供 iOS 和 Android 原生应用。

#### (2) 业务逻辑层

##### ① 核心服务
实现核心业务逻辑。

##### ② 数据访问
封装数据库操作。
```

## 附加规则

1. **不允许超出以上四个层级**
   - 文档中不得出现 Level 5 或更深的标题层级

2. **每个层级的标题编号必须按上述格式编写**
   - 不得使用自定义的编号格式
   - 不得遗漏编号前缀或空格

3. **编号必须连续且正确**
   - Level 1：1, 2, 3, ...
   - Level 2：1.1, 1.2, 2.1, 2.2, ...
   - Level 3：(1), (2), (3), ...
   - Level 4：①, ②, ③, ...

4. **标题层级关系必须正确**
   - Level 2 必须属于某个 Level 1
   - Level 3 必须属于某个 Level 2
   - Level 4 必须属于某个 Level 3

---

## 设计动机

统一的标题层级保证改写文档的结构一致性：

1. **并行处理支持**：Skill 的改写流程会将文档拆分到多个 Sub Agent 并行处理，只有严格的标题规范才能确保各章节独立改写后仍能无缝拼合

2. **避免层级冲突**：统一的标题规范避免出现层级冲突或编号错乱

3. **结构清晰**：四级标题体系为大多数技术文档提供了足够的层次结构

---

## 注意事项

1. 开发者在实现标题编号逻辑时应**逐字段对照**规格定义
2. 标题内容应简洁明了，准确反映该章节的主题
3. 避免过深的层级嵌套，保持文档的可读性
4. 如果文档内容复杂，考虑调整内容组织方式，而不是增加层级
